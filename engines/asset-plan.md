# 资产规划引擎

根据视频类型和复杂度，自动决定需要生成哪些参考资产。含最低资产硬性要求。

---

## 一、按视频类型的最低资产

### 类型匹配（从 story-intake 的类型 → 资产类型）

| story-intake 类型 | → 资产类型 | 条件 |
|------------------|-----------|------|
| 武侠/战争 | 战斗/动作 | 有打斗描述；否则 → 双人对峙 |
| 科幻 | 科幻/机甲 | 有机甲/装备描述；否则按人数匹配 |
| 玄幻 | 大场景神话 | 有神魔/巨物；否则按人数匹配 |
| 爱情 | 双人对峙/对话 | 默认；单角色 → 单人情绪戏 |
| 恐怖/悬疑 | 单人情绪戏 | 默认；多角色 → 双人对峙 |
| 喜剧/青春 | 单人情绪戏 | 根据场景调整 |
| 末日 | 大场景神话 | 有宏大场景；否则按人数匹配 |
| 剧情（默认） | 按实际角色+场景内容匹配 | 单角色→单人 / 双角色→双人 / 多角色→群像 / 有战斗→战斗 |

### 单人情绪戏
```
☑ 角色卡 ×1（LS11 干净一致性卡）
☑ 首帧 ×1 + 尾帧 ×1
不满足 → 禁止输出视频 prompt
```

### 双人对峙/对话
```
☑ 角色卡 ×2（双方各自）
☑ 场景空间关系图 ×1（LS24 双人关系板）
☑ 首帧 ×1
不满足 → 禁止输出视频 prompt
```

### 战斗/动作
```
☑ 角色卡 ×1（主角）
☑ 武器/道具卡 ×1（如有特殊武器）
☑ 动作关键帧 ×3（起势/高潮/收势）
☑ 首帧 ×1 + 尾帧 ×1
不满足 → 禁止输出视频 prompt
```

### 大场景神话/奇幻
```
☑ 场景全景 ×1（LS8 场景全能参考板）
☑ 角色卡 ×1（比例衬托）
☑ 全案分镜图 ×1（LS6 横幅时间轴）
不满足 → 禁止输出视频 prompt
```

### 多角色群像（≥3人）
```
☑ 角色卡 ×N（每个主要角色）
☑ 站位俯视图 ×1（LS27 场景蓝图俯视板）
☑ 全案分镜图 ×1（LS15 导演作战室）
不满足 → 禁止输出视频 prompt
```

### 科幻/机甲
```
☑ 角色卡 ×1（LS13 科幻神使系统卡）
☑ 机甲/装备拆解卡 ×1（LS22 服装层次爆炸图）
☑ 场景图 ×1（科幻环境）
不满足 → 禁止输出视频 prompt
```

### 神兽/生物
```
☑ 生物角色卡 ×1 + 比例尺 ×1 + 场景图 ×1
不满足 → 禁止输出视频 prompt
```

### 追车/追逐
```
☑ 场景路线图 ×1 + 角色卡 ×1 + 关键帧 ×2
不满足 → 禁止输出视频 prompt
```

## 二、预算控制

| 级别 | 参考图数量 | 适用场景 |
|------|----------|---------|
| 极简 | 2-3张 | 快速验证、简单情绪戏 |
| 标准 | 3-5张 | 大多数商业片、短剧 |
| 精细 | 5-8张 | 复杂场景、高一致性要求 |
| 电影级 | 8-12张 | 长片、多角色、大预算 |

---

## 三、生成顺序

```
必须先出（锁身份和空间）：
  1. 角色卡（锁谁）
  2. 场景全景（锁在哪）

然后出（锁动作和关系）：
  3. 关键动作帧（锁怎么动）
  4. 空间关系图（锁谁和谁什么关系）

最后出（锁起点终点）：
  5. 首帧（镜头1）
  6. 尾帧（最后镜头）
```

---

## 四、资产复用

| 资产 | 可复用于 |
|------|---------|
| 角色卡 | 所有含该角色的镜头 |
| 场景全景 | 同一场景的所有镜头 |
| 道具卡 | 道具出现的所有镜头 |
| 首帧 | 也作为视频首帧锚点 |
| 尾帧 | 也作为下段首帧（跨段衔接） |

---

## 五、资产质量检查

| 检查项 | 要求 |
|--------|------|
| 角色卡 | 五官/发型/服装/体型不可跨镜变化 |
| 场景图 | 主光源方向明确、空间关系清晰 |
| 首尾帧 | 同一角色、同一场景、同一色调 |
| 关键动作帧 | 运动方向明确、动作可执行 |
| 空间关系图 | 角色站位、间距、视线方向清楚 |

## 资产变更传播

当角色卡或场景图重新生成（immutable_features / fixed_elements 变更）时，**必须**通过 project-graph 向下游传播：

```
资产变更（variable-registry 中的 immutable_features 或 fixed_elements 更新）
  ↓
1. 调用 engines/incremental-update.md：
   ├─ entity_type = character（角色卡变更）或 scene（场景图变更）
   ├─ 查询 state/project-graph.md → affected_shots / affected_assets
   └─ 获得精确的影响范围（不再手动扫描全部 shot-state）
2. 更新受影响镜头：
   ├─ 角色变更 → 更新各镜 action 中的外观描述
   ├─ 场景变更 → 更新各镜 lighting/color 中的空间约束
   └─ 同步标记 asset-map 中需要重新生成的 @图
3. 触发 consistency-engine（增量模式）：
   └─ 只评估受影响的 RM 维度
4. 输出变更影响报告
```

### 示例

```
重新生成角色卡 "墨渊" → immutable_features 更新：
  旧：黑衣/长发/暗金护腕
  新：黑衣/长发/暗金护腕/左脸疤痕/「断念」剑身裂纹

project-graph 查询：
  affected_shots("character", "C1") → [SH01, SH03, SH05, SH07]
  affected_assets("character", "C1") → [@图1]

传播：
  → SH1 action 追加 "left cheek scar visible"
  → SH3 action "剑身细密裂纹泛青光"
  → SH5 action "疤痕在闪电下格外醒目"
  → SH7 尾帧 "裂纹沿剑身蔓延至剑尖"
  → @图1 标记「⚠ 待重新生成」

受影响: 4镜（SH1/3/5/7）| 跳过: 3镜（SH2/4/6）| 重评: 仅 Character RM
```

## 联动

← 读取 `state/platform-config.md`（段数 → 全案图：每段1张。禁止60s只出1张全案图）
← 接收 `video-director` 的参考图决策 + `story-intake` 的视频类型
→ 输出资产列表给模板引擎
→ 不满足最低资产 → `auto-repair` 触发修复
→ 满足 → 放行到 `reference-anchor`
→ **写入 `state/variable-registry.md`**（style.layout_*, characters.*.immutable_features, scene.primary.fixed_elements）
→ **资产变更时触发传播**：调用 `engines/incremental-update.md` → 查询 `state/project-graph.md` → 更新关联镜头 → 增量重评 consistency-engine
