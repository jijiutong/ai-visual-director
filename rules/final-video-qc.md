# 视频最终质检规则

生成视频prompt前的最后一道阀门。全项通过才放行。

---

## 质检清单

### 1. 时长一致性

```
□ 宣告时长 = 各镜时长之和？
□ 各镜时长在平台限制内？
□ 有没有某镜过短（<0.5s AI做不了）？
□ 有没有某镜过长（>8s 视频呆板）？
```

### 2. 角色一致性

```
□ 所有镜中同一角色的五官/发型/服装一致？
□ 角色DNA的3个不可变特征每镜都声明？
□ 多角色时：身高差/体型差/肤色差每镜一致？
□ 跨镜没有"角色换人"现象？
```

### 3. 场景一致性

```
□ 同一场景的主光源方向每镜一致？
□ 同一场景的主色调每镜一致？
□ 同一场景的空间关系（门/窗/家具位置）每镜一致？
□ 跨镜没有"场景漂移"？
```

### 4. 动作可执行性

```
□ 每镜运动组合 ≤ 3个？（motion-physics）
□ 运动方向逐镜声明并连续？
□ 人物+镜头+环境运动不冲突？（motion-physics）
□ 高潮镜的运动预算最充足？
```

### 5. 平台兼容性

```
□ Prompt字数在平台限制内？
□ 语言符合平台要求？
□ 参考图数量在平台限制内？
□ 时长在平台限制内？
```

### 6. 连续性与状态继承

```
□ 每镜结束状态已记录？
□ 下镜起始状态 = 上镜结束状态？
□ 场景切换有明确转场过渡？
□ 如有拆段：尾帧→首帧已衔接？
```

### 7. Prompt格式

```
□ 中文无乱码？
□ 英文语法正确？
□ 负面提示词与平台匹配？
□ 特殊参数正确（--cref / first_frame等）？
```

### 8. 引用一致性（新增）

```
□ 视频 prompt 中 @图引用数量 ≤ asset-map 已声明的图片数？
□ 视频 prompt 中引用的镜头编号全部在 shot-state 中存在？
□ 台词引用的 shot_id 与 dialogue-map 一致？
□ storyboard 引用图片与 asset-map 中 type=storyboard_board 的 @编号一致？
□ 角色引用图片与 asset-map 中 type=character_sheet 的 @编号一致？
□ 场景引用图片与 asset-map 中 type=scene_reference 的 @编号一致？
```

### 9. 模板合规（新增）

```
□ 角色卡 prompt 是否由 /character 子路由 + templates/character-sheet.md 生成？（禁止 video-prompt-assembly 手写角色 prompt）
□ 场景图 prompt 是否由 /scene 子路由 + templates/scene-card.md 生成？
□ 分镜图 prompt 是否由 /storyboard 子路由 + templates/full-board.md 生成？
□ 输出中是否包含正确的编号引用（VS/EC/CN/EV/WT/MT/CP/ME/BL/HE/SD/PR/CR/DR）？
□ 全案板是否按段独立（每段一张，段长由 {PLATFORM}_MAX_DURATION 决定，非一张塞全部镜）？
```

### 10. 视觉干净度（新增）

```
□ 输出类型的视觉密度是否在允许范围（参考 rules/visual-cleanliness.md 默认密度表）？
□ 背景是否抢主体？（background_complexity 不得高于输出类型上限）
□ 是否有随机网格、伪 UI、无意义微文字？（角色卡/三视图/视频首帧禁止）
□ 粒子、烟雾、纹理是否过量？
□ 视频首帧/角色卡是否足够干净？（致命项：角色卡背景太复杂 / 视频首帧大量文字HUD）
□ HUD 是否只出现在允许的科技类模板中？（非科技类禁止 HUD）
```

### 11. 视觉可用性（新增）

> 检查产出图是否适合其用途。display_asset 误用作 video_asset 是常见致命错误。

```
□ 是否有大量可见文字？（含乱码/伪文字/背景文字）→ video_asset 必须无
□ 是否有边框/表格/UI 框架？（分镜格线/信息栏/参数表）→ video_asset 必须无
□ 是否有伪 HUD / 随机科技线框？→ 非 LS13 科技卡必须无
□ 是否粒子/雾效/光点过多淹没主体？→ density ≥ 4 才允许 heavy
□ 是否主体被背景抢走？（主体占画面 < 40%）→ video_asset 主体必须 ≥ 50%
□ 是否适合作为 AI 视频模型参考图？（video_safe 判定）
□ 是否 display_asset / marketing_asset 被误放入视频 @图引用？
```

**用途误用检测**（检查 `state/asset-map.md`）：

```
读取 asset-map → 遍历视频 @图中的 asset_purpose：
  ├─ asset_purpose = display_asset → ⚠ 致命：display 资产被误用为视频参考
  │   触发：阻断 → 建议用 /declutter video-ref 派生 clean 版本
  │   或：提示用户生成 video_asset（clean 角色卡/clean 场景锚点）
  ├─ asset_purpose = marketing_asset → ❌ 致命：海报/封面不应进视频流程
  │   触发：阻断 → 从 @图列表中移除
  ├─ asset_purpose = video_asset → ✅ 放行
  └─ asset_purpose = consistency_asset → ✅ 放行（如 LS9 九宫格空间锁定）
```

**致命/非致命判定**（参考 `rules/visual-cleanliness.md` 四）：

| 问题 | 级别 |
|------|------|
| 角色卡背景太复杂 | **致命** |
| 视频首帧大量文字/HUD | **致命** |
| 三视图有背景/噪点 | **致命** |
| display_asset 被误用为 video_asset | **致命** |
| marketing_asset 出现在视频 @图 | **致命** |
| video_asset 有明显文字/边框 | **致命** |
| video_asset 主体 < 50% | **致命** |
| 海报粒子稍多 | 非致命 |
| 世界观板信息多但分区清楚 | 放行 |
| display_asset 文字/标注合理（用作展示） | 放行 |

### 12. 台词连贯性 + 音效覆盖（新增）

> 检查台词和音效是否覆盖了所有需要的关键镜头。

```
□ 每句台词的 shot_id 是否在 shot-state 中存在？（引用完整性）
□ 每句台词的 speaker 是否在 variable-registry 角色列表中存在？
□ 相邻台词之间有情绪/内容承接？（不凭空跳话题）
□ 沉默设计是否有动机？（不无故冷场 ≥3s）
□ 台词量是否压过画面表现力？（动作高潮镜无冗余台词）
□ 口型同步标记是否正确？（近景/特写=✅，远景/背面=❌）
□ 音效是否覆盖全部镜头？（每镜至少 1 条环境音或音乐）
□ 拟音是否与镜头动作匹配？（动作镜必须有对应 FX）
□ 音乐情绪曲线是否与 EC 编号一致？
□ 混响空间是否与场景空间匹配？
□ 音效与台词时机是否冲突？（音效高潮不与台词重叠）
```

**致命/非致命判定**：

| 问题 | 级别 |
|------|------|
| 台词 shot_id 不存在于 shot-state | **致命** |
| 台词 speaker 不存在于角色列表 | **致命** |
| 动作高潮镜无对应拟音 FX | **致命** |
| 音乐情绪与 EC 编号矛盾 | **致命** |
| 音效与台词高峰期重叠 | 非致命 |
| 个别镜音乐缺失（但环境音覆盖） | 非致命 |
| 沉默设计缺乏动机 | 非致命 |

## 质检结论

| 结论 | 条件 | 操作 |
|------|------|------|
| 通过 | 全部12项 ✅ | 放行，输出 render-package |
| 有条件通过 | 1-2项 ⚠（非致命） | 标记问题+建议，放行 |
| 不通过 | 任何 ❌（致命）或 ≥3项 ⚠ | 触发 auto-repair 修复后重新质检 |

---

## 致命项 vs 非致命项

| 致命（不通过） | 非致命（标记但放行） |
|-------------|----------------|
| 字数严重超限（>150%） | 字数轻微超限（<110%） |
| 角色DNA声明缺失 | 角色DNA声明不完整 |
| 平台完全不兼容（中文发英文平台） | Prompt格式小问题 |
| 运动中存在逻辑矛盾 | 运动虽然不打架但不够细致 |
| @图引用缺失/错位（引用数量≠asset-map声明） | @图引用描述与asset-map不完全一致 |

## 联动

← 接收所有引擎和模板的输出
← **读取 `state/asset-map.md`** 校验 @图引用一致性
← **读取 `state/shot-state.md`** 校验镜头编号引用
← **读取 `state/dialogue-map.md`** 校验台词镜号
→ 通过 → `render-package` 输出
→ 不通过 → `auto-repair` 修复
→ 修复后重新质检（最多 REPAIR_MAX_ROUNDS 轮，从 api-config.template.env 读取）
→ REPAIR_MAX_ROUNDS 轮后仍不通过 → 标记为"需人工介入"
